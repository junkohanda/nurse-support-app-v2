# 看護師サポートアプリ v2

## なぜ作ったか

夜勤明けの看護師が、5つも6つもアプリを開く余力はない。

看護師の勤務後は身体的・精神的な疲労が極限に近い状態にある。「今日の気分を記録したい」「明日のシフトを確認したい」「覚えた医療用語をメモしておきたい」——これらはすべて別々のアプリに散らばっており、疲れた状態でアプリを使い分けること自体が負担になっていた。

15年以上の臨床経験と10年以上のマネジメント経験を持つ元看護師として、この課題を肌で知っていた。だから「必要な機能をすべて一つにまとめる」というコンセプトでアプリを設計した。

---

## 機能一覧

| 機能 | 概要 |
|------|------|
| 日記 + 気分ログ | 毎日の記録を残す。気分の変化を数値・絵文字で記録し、メンタルの傾向把握にも使える |
| シフト管理 | カレンダー表示・連続入力・画像シェア対応。看護師特有の不規則シフトに対応 |
| ToDo | 優先度・期日の設定が可能。業務上のタスクも私生活のタスクも一元管理 |
| 勉強ノート | カテゴリ分類・復習日設定付き。資格勉強や看護技術の復習を習慣化しやすくした |
| 医療用語辞典 | 現場で覚えた用語をその場で登録・検索できる個人用辞典 |
| 設定（カスタムシフト） | 職場ごとに異なるシフト名称を自分で定義できる |

---

## 開発プロセス（AI活用開発）

このアプリは**AI活用開発**の手法で制作した。設計・要件定義・機能の優先順位付けはJunkoが担当し、コードの実装はAI（Claude）が担当している。

「自分で書けるから作った」ではなく、「現場の課題を知っているから設計できた」というアプローチをとっている。技術的な実装を外部（AI）に委ねることで、自分の強みである課題発見・要件定義・UX設計に集中した。

**技術スタック（AI活用開発）**

- フロントエンド: Vite + Tailwind CSS
- バックエンド: Supabase（認証・クラウドDB）
- 開発補助: Claude（コード実装）

---

## v1 から v2 への進化

| 項目 | v1 | v2 |
|------|----|----|
| データ保存 | ブラウザのローカルストレージ | Supabase（クラウドDB） |
| 認証 | なし | Supabase Auth（メール認証） |
| 端末をまたいだ利用 | 不可 | 可能 |
| データの永続性 | ブラウザ削除で消える | クラウドに保存 |

v1はプロトタイプとして機能検証を行い、「使えるもの」の感触をつかんだ。v2ではそれを実用レベルに引き上げるため、認証とクラウドDBの導入を決めた。

---

## AIコードレビューによる品質改善

実装後、Codex（OpenAIのコードレビューAI）を使って品質チェックを実施した。その結果、8つの問題が発見され、すべて修正した。

**発見・修正した主な問題**

- DBスキーマの設計ミス（テーブル構造の修正）
- エラーハンドリングの不足（API通信の失敗時対応の追加）
- タイムゾーン処理の誤り（シフト日付のズレを修正）
- コンポーネントの過剰な肥大化（責務を分割してメンテナンス性を向上）

「AIが書いたコードだから品質は大丈夫」とは考えず、別のAIツールでレビューをかけるという工程を設けた。AIを使いこなすうえでの批判的な視点も、自分の役割として持っている。

---

## この開発で得た学び

- 要件定義の粒度が実装品質に直結する（曖昧な指示はバグの温床になる）
- ユーザー（看護師）視点の課題定義があってはじめて、AIへの指示が具体的になる
- AIコードは動くが、設計の甘さはAIでは補えない

## ポートフォリオカード用説明文

勤務後の極度の疲労状態にある看護師が、複数のアプリを使い分けることなく日常を記録・管理できるよう設計した統合アプリ。日記・気分ログ・シフト管理・ToDo・勉強ノート・医療用語辞典を一つにまとめた。設計・要件定義は元看護師として担当し、実装はAI（Claude）を活用。v2ではSupabase認証・クラウドDBを導入し、実用レベルに進化させた。AIコードレビューで8つの問題を発見・修正し、品質担保の工程も設けた。
